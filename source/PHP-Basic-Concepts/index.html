Compiled vs. Interpreted Languages

Programming work-flow

Compiled
  • Write human-readable source code
  • Use a compiler to translate source code into machine code
  • Correct compile-time errors
  • Execute machine code with a debugger
  • Correct run-time errors
  • Repeat until successful…
Interpreted
  • Write human-readable source code
  • Execute source code for debugging with an interpreter
  • Correct syntax & run-time errors
  • Repeat until successful…
Comparing the programming work-flow. (0) In a compiled work-flow: (1) You first write source code in a human readable format, (2) Then using a compiler, you generate machine code from the source code. (3) If any errors occur during compile, review & correct them. (4) Once the code compiles without errors, you execute the generated code with a debugger. (5) If any errors present themselves during the run, review & correct them; & finally, (6) Repeat these steps until all errors are addressed. (7) In an interpreted work-flow: (8) You start off like before, writing human readable source code, however you go directly into (9) executing the code with an interpreter for debugging, skipping the compile & review steps. (10) If any errors present themselves during the run, review & correct them; & again, (11) Repeat these steps until all errors are addressed.

PHP Input & Output

Processing the stream of data

PHP is an interpreted language, not a compiled one:

  • Takes input (file or stream) containing text and/or PHP instructions
  • Creates output (stream), most commonly HTML

Since PHP 4: The PHP parser preprocesses the input to produce opcodes for processing by the Zend Engine.

Processing the stream of data. (0) PHP is an interpreted language, not a compiled one. (1) PHP takes input containing text & PHP instructions, (2) Then creates output, typically HTML. (3) Since PHP 4: the parser preprocesses the input to produce opcodes for the Zend Engine.

Preprocessing

Human-readable code to executing program

  1. Preprocessor
    1. Lexical Analyzer (lexer)
      1. Reads in the human-readable source code
      2. Interprets the source code into machine-digestible tokens
      3. Passes the tokens to the Parser
    2. Parser
      1. Creates an operation code array (opcode / op array) from the tokens
        – The op array is also known as an abstract syntax tree (AST)
      2. Passes the op array to the Executor
  2. Executor
    1. Steps through the op array
    2. Executing each opcode in the op array with the Zend Engine
Taking a PHP script from human-readable code to execution. First the script gets consumed by, (0) The preprocessor & then is passed to (1) The executor. The preprocessor consists of (2) A lexer & (3) A parser. The lexer (4) Reads the source code, (5) Interprets the code into tokens, (6) Then passes the tokens to the parser. The parser (7) Creates an operation code array (known as opcodes, or an op array) from the tokens (8) The op array is also know as an abstract syntax tree (9) Then the op array is passed to the executer. The executor (12) Iterates threw the op array & (13) Executes each opcode with the Zend Engine.

Zend Engine (v2.0)

The Zend Engine simulates an ideal computer. The VM architecture provides hardware independent flexibility & portability. While a standard (x86) computer processor has 75 base operations, the Zend Engine has approximately 150 instruction (opcodes).

The Virtual Machine

  • Keeps track of requests
  • Processes scripting files
  • Handles variables & resources

The VM opcode set includes simple logic & math operations, as well as complex operations such as include(). A VM is always slower than the host machine; some speed penalty is regained by performing complex instructions as a single VM operation.

Version 2.0 of the Zend Engine (0) Simulates an ideal computer environment. (1) The architecture of the Virtual Machine provides flexibility & portability independent of the hardware. (2) An x86 processor has 75 base operations; the Zend Engine has ~150 instructions (or opcodes). (3) The Virtual Machine tracks requests, (4) Processes scripts, & (5) Handles variables & resources. (6) The opcode set includes simple logic & math operations, as well as complex operations like `include()`. (7) A Virtual Machine is slower than its host, some speed is regained threw performing complex instructions as single operation.

PHP Tags

Opening & Closing tags

  1. <?php … ?> Always Available
  2. <script language="php"> … </script> Always Available
  3. <? … ?> Short Style
  4. <?= … ?> Echo - Short Style
  5. <% … %> ASP Style
  6. <%= … %> Echo - ASP Style

The Opening tag tells the interpreter to turn on

The Closing tag tells the interpreter to turn off

The Echo tags are short for <?php echo; ?>

PHP opening & closing tags tell the Zend Engine what's to be interpreted & what's not. There is a standard set, but like all things PHP, you have options. (0) Tag sets 1 & 2 are (1) Always available; the first set is most common & recommended. Other tag sets should be avoided when developing for distribution, to avoid inconsistency across environments. The remaining tag sets require changes to the PHP configuration file to enable them. (2) Tag sets 3 & 4 are (3) Short style tags; to enable them, the configuration directive is short_open_tag. Since PHP version 5.4, the parser recognizes these sets regardless of the configuration. (4) Tag sets 5 & 6 are (5) Active Server Page style tags; to enable them, the configuration directive is asp_tags. (6) The opening tag tells the interpreter to turn on, & (7) The closing tag tells the interpreter to turn off. (8) Tag sets 4 & 6 are echo tags, a convenience so you don't have to explicitly call echo. Since PHP version 5.3, the parser won't throw an error if it consumes a file with only an open tag with na minimum of one trailing white space character. A closing tag at the end of a file is optional. If you deal with return headers, leaving it off is helpful, so unwanted white space doesn't cause errors.

Escaping

Switching the interpreter on & off

Everything outside of the PHP tags is escaped.


 <html>
     <body>
         <h1><?php echo $title; ?></h1>
     </body>
 </html>

Escaped content is passed threw as output.


 <html>
     <body>
         <h1>Escaping PHP</h1>
     </body>
 </html>
Switching the interpreter on & off. (0) Anything and everything outside of the PHP tags is escaped, (1) Escaped content passes threw as output. (2) As you see, all content not inside the PHP tags is output unaffected, & (3) Content inside the tags is processed and the result is output.

Escaping


 <html>
     <body>
         <h1><?php echo $title; ?>
         </h1>
     </body>
 </html>

A new-line character directly after a closing tag is ignored.


 <html>
     <body>
         <h1>Escaping PHP</h1>
     </body>
 </html>
(0) New-line characters following directly after a closing tag is ignored and not output. (1) As you can see the ending H1 tag is now on the same line as its beginning tag.

Escaping

Escaping large blocks of output is more efficient than using an output function or command, such as echo or print.

Parsing in this manner allows PHP to be embedded in all sorts of different documents.

  • HTML
  • CSS
  • SVG
  • CSV
  • XML
  • JS
Using the escape method for outputting large blocks of content is more efficient than using a PHP function or command like echo or print. (0) Parsing content in this manner allows PHP to be embedded with ease in many types of text files like: (1) Hypertext Markup Language, (2) Extensible Markup Language, (3) Scalier Vector Graphics, (4) Cascading Style Sheets, (5) JavaScript, & (6) Comma Separated Value files.
